# Original Language Data Package Plan

## Status

Design approved for documentation and Git protection policy only. Exporter, verifier, importer, schema changes, database writes, and migration tracking tables are not implemented by this document.

## Purpose

Word Covenant Ministry needs an official Data Package system for original-language data so another Local/Flywheel environment can restore the same reviewed original-language dataset without relying on SQL dumps.

The package must preserve the source original-language corpus, reviewed Korean presentation fields, and enough verification metadata to detect incomplete, corrupted, or incompatible imports before any database write occurs.

This system is for controlled project data movement. It is not a public bulk API, not a frontend payload, and not a replacement for schema migrations.

## JSONL File Specification

JSONL is required so large datasets can be streamed line by line without loading the full corpus into memory.

Rules:

- One valid JSON object per line.
- UTF-8 encoding.
- No trailing comma.
- No SQL statements.
- No WordPress serialized payloads.
- Deterministic export ordering.
- Empty strings and nulls must preserve database meaning.

### original-terms.jsonl

Each record represents one row from `wcm_original_terms`.

Required fields:

```json
{
  "id": 7,
  "term_identity_hash": "<sha256>",
  "language_type": "hebrew",
  "lemma": "אֱלֹהִים",
  "lemma_normalized": "אלהים",
  "strongs_number": "H430",
  "strongs_extended": "H430",
  "transliteration": "'E.lo.Him",
  "transliteration_ko": "엘로힘",
  "gloss": "God",
  "gloss_ko": "하나님",
  "created_at": "<db-value>",
  "updated_at": "<db-value>"
}
```

Identity rule:

- `term_identity_hash` is the authoritative restore key.
- Package `id` is retained for occurrence mapping, but cross-environment matching must use `term_identity_hash`.

### original-word-occurrences.jsonl

Each record represents one row from `wcm_original_word_occurrences`.

Required fields:

```json
{
  "id": 1,
  "term_id": 7,
  "term_identity_hash": "<matching-term-hash>",
  "book_id": 1,
  "chapter": 1,
  "verse": 1,
  "word_order": 3,
  "subword_order": 0,
  "token_type": "word",
  "surface_form": "אֱלֹהִים",
  "normalized_form": "אלהים",
  "morphology": "Ncmpa",
  "grammar_summary": null,
  "grammar_note": null,
  "contextual_function": null,
  "source_dataset": "STEP_TAHOT",
  "source_ref": "Gen.1.1"
}
```

Occurrence identity rule:

```txt
source_dataset + book_id + chapter + verse + word_order + subword_order + token_type
```

Importers must use this identity to avoid duplicate occurrence rows.

## Checksum Policy

The exporter must create SHA-256 checksums for every generated data file.

Required file:

```txt
checksums.sha256
```

Required behavior:

- Include `original-terms.jsonl` checksum.
- Include `original-word-occurrences.jsonl` checksum.
- Include checksums in `manifest.json` as well.
- Verifier must fail if any checksum differs.
- Generated checksum files for local exports may use `.sha256.generated` and must not be committed.

## expected_count Verification

Every package must include expected counts in both the manifest and per-file metadata.

Verifier must check:

- `original-terms.jsonl` line count equals manifest expected count.
- `original-word-occurrences.jsonl` line count equals manifest expected count.
- Terms count equals current verified export target when exporting from this project state: `16891`.
- Occurrences count equals current verified export target when exporting from this project state: `673263`.
- Source dataset counts match manifest values.

Import must stop before database writes if counts do not match.

## Verify Workflow

Planned verifier tool:

```txt
backend/app/public/wp-content/plugins/wcm-core/tools/verify-original-language-package.php
```

Required checks:

1. Manifest exists and is valid JSON.
2. Package type is `original_language`.
3. `db_version` is compatible with current `wcm_core_db_version`.
4. Required files exist.
5. SHA-256 checksums match.
6. JSONL lines are valid JSON.
7. Required fields are present.
8. Expected counts match actual line counts.
9. No duplicate `term_identity_hash` values exist.
10. No duplicate occurrence identity values exist.
11. Every occurrence `term_identity_hash` resolves to a packaged term.
12. Source datasets are expected values such as `STEP_TAHOT` and `STEP_TAGNT`.

Verifier must be read-only.

## Dry-run / Apply / Rollback Workflow

### Dry-run

Importer must default to dry-run.

Planned command shape:

```bash
php tools/import-original-language-package.php --package=/path/to/package
```

Dry-run must:

- Run full verifier checks.
- Compare package terms to local terms by `term_identity_hash`.
- Report inserts, matches, skipped records, and conflicts.
- Detect conflicting reviewed Korean fields.
- Validate expected occurrence identity behavior.
- Produce an import plan.
- Write no database changes.

### Apply

Database writes require explicit `--apply`.

Planned command shape:

```bash
php tools/import-original-language-package.php --package=/path/to/package --apply
```

Apply must:

- Re-run verification immediately before writes.
- Import/update terms before occurrences.
- Build a package-term-id to local-term-id map through `term_identity_hash`.
- Preserve existing reviewed Korean fields.
- Insert missing occurrences only when identity is absent.
- Avoid duplicate rows.
- Use chunked writes to avoid memory and timeout issues.
- Record import manifest details after successful apply once tracking storage is implemented.

### Rollback

Rollback must be planned before apply.

Accepted rollback strategies:

1. Database backup before apply.
2. Row-level prior-value backup file generated before apply.
3. Future `wcm_seed_migration_rows` or equivalent tracking table.

Rollback requirements:

- Code rollback uses previous Git tag/plugin version.
- Schema rollback requires explicit migration plan; automatic destructive schema rollback is not allowed.
- Data rollback uses DB backup or stored prior row values.
- Importer must not claim rollback support unless a backup or prior-value record exists.

## Reviewed Korean Field Protection

Importer must not overwrite reviewed Korean values.

Protected fields:

```txt
transliteration_ko
gloss_ko
```

Rules:

- If local value is null/empty and package value is present, importer may plan a fill.
- If local value equals package value, importer may treat it as unchanged.
- If local value exists and package value differs, importer must report a conflict and skip by default.
- Any overwrite mode requires a future explicit option and separate approval.
- English `transliteration` and `gloss` must remain unchanged unless a future approved source migration explicitly changes them.

## Other Local/Flywheel Setup Procedure

Target restore procedure for another Local/Flywheel environment:

1. Clone repository.
2. Check out the intended Git tag or commit.
3. Confirm `.gitignore` includes generated data protections before any export/import work.
4. Install backend plugin dependencies if needed.
5. Confirm Local WP site uses the official plugin path:

```txt
backend/app/public/wp-content/plugins/wcm-core/
```

6. Confirm `wcm_core_db_version` is compatible with package manifest `db_version`.
7. Store downloaded Data Packages outside Git, or under an ignored import path such as:

```txt
storage/imports/original-language/<package-id>/
```

8. Run package verifier.
9. Run importer dry-run.
10. Review conflicts and expected counts.
11. Apply only with `--apply` after dry-run passes.
12. Do not commit generated import/export files after apply.
13. If `git status` shows `.sql`, full `.jsonl`, `.zip`, `.tar`, or `.gz` files, stop before committing and either remove the generated file from the worktree or confirm ignore policy.
14. Run API smoke checks for Genesis 1, Psalms 1, Matthew 1, and John 1.
15. Confirm Korean fields display in frontend after normal frontend validation.

## Personal Local Ignore Policy

Developers may also use `.git/info/exclude` for machine-specific generated paths that should not be shared with the whole team.

Use `.git/info/exclude` for personal-only paths such as:

```txt
local-original-language-package/
tmp-original-language-export/
flywheel-downloads/
```

Do not use `.git/info/exclude` as a substitute for project-wide protections. Any path that future developers are likely to generate, such as Data Package exports, imports, backups, SQL dumps, or generated JSONL files, belongs in `.gitignore`.

## Local pre-commit Hook Guard

Developers should install a local pre-commit hook before export/import work. This is a local safety guard and is not yet a CI requirement.

Example `.git/hooks/pre-commit`:

```sh
#!/bin/sh
set -eu

blocked_files=$(git diff --cached --name-only --diff-filter=ACMR | awk '
  /\.sql$/ { print }
  /\.zip$/ { print }
  /\.tar$/ { print }
  /\.gz$/ { print }
  /^docs\/data-packages\/.*\.jsonl$/ && $0 !~ /\/sample-[^\/]*\.jsonl$/ { print }
  /^docs\/data-packages\/.*\.sha256\.generated$/ { print }
')

if [ -n \"$blocked_files\" ]; then
  echo \"Blocked generated data files from commit:\" >&2
  echo \"$blocked_files\" >&2
  echo \"Move package artifacts to storage/imports or storage/exports, or commit only README.md, manifest.example.json, and sample-*.jsonl.\" >&2
  exit 1
fi

large_jsonl=$(git diff --cached --name-only --diff-filter=ACMR -- '*.jsonl' | while read -r file; do
  [ -f \"$file\" ] || continue
  case \"$file\" in
    docs/data-packages/*/sample-*.jsonl) continue ;;
  esac
  bytes=$(wc -c < \"$file\" | tr -d ' ')
  if [ \"$bytes\" -gt 1048576 ]; then
    echo \"$file ($bytes bytes)\"
  fi
done)

if [ -n \"$large_jsonl\" ]; then
  echo \"Blocked large JSONL files from commit:\" >&2
  echo \"$large_jsonl\" >&2
  exit 1
fi
```

After creating the hook, make it executable:

```bash
chmod +x .git/hooks/pre-commit
```

The hook should block:

- `.sql`
- Large `.jsonl`
- `.zip`
- `.tar`
- `.gz`
- Generated data under `docs/data-packages/`

The hook should allow:

- `docs/data-packages/**/README.md`
- `docs/data-packages/**/manifest.example.json`
- `docs/data-packages/**/sample-*.jsonl`
- Markdown and other documentation files

## Future CI Guard

CI should later add a repository-level generated data guard so GitHub rejects accidental data package commits even when a developer has no local hook installed.

Future CI checks should fail pull requests that include:

- SQL dumps.
- Full original-language JSONL exports.
- Archive files such as `.zip`, `.tar`, or `.gz`.
- Generated package files under `docs/data-packages/` except approved examples.
- Oversized files above the documented project threshold.

This CI guard is not implemented in the current documentation-only phase.
